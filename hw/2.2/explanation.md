### 2.2a
Пусть потоки A и B в critical_section(), то есть в A.4 и B.6 соответственно.  
Чтобы это произошло потокам необходимо выйти из их while'ов(A.2 и B.2) по concurrent execution trace.  
Получается, что условия в A.2 и B.2 на данный момент false.  
То есть A_flag == false и B_flag == false.  
Перед A.2 и B.2 должны выполниться A.1 и B.1(поднятия флагов) по concurrent execution trace.  
То есть перед A.4 был поднят A_flag, а перед B.6 был поднят B_flag.  
A.1, A.2, B.1, B.2 атомарные операции.  

Допустим, что A.4 достигнута раньше B.6.  
Тогда A_flag поднят, но B.6 требует обратное.  
Самостоятельно поток B не может его опустить.  
Перед этим должно произойти A.5, но тогда A выйдет из critical_section().  
Но после возвращения в critical_section() снова поднимется флаг и тд.  
Значит B.6 достигнута раньше A.4?  

Аналогичная ситуация!  
B_flag поднят, но A.4 требует обратное.  
Самостоятельно поток A не может опустить B_flag.

### 2.2b
Поток A может зациклиться только в A.2.  
Поток B может зациклиться только в B.2 и B.4.  
  
Для того, чтобы поток B зациклился в B.2 требуется A_flag == true.  
То есть чтобы A.5(единственное опускание A_flag в коде) не выполнялось.  
Для этого нужно чтобы A зациклился в A.2.  
То есть B_flag == true.  
Перед A.2 следует A.1 - поднятие A_flag по concurrent execution trace.    
Перед A.2 следует B.1 - поднятие B_flag для зацикливания.  
Перед B.2 следует A.1 - поднятие A_flag для зацикливания.  
Перед B.2 следует B.1 - поднятие B_flag по concurrent execution trace.  
Допустим A.2 зациклился перед B.2.  
Тогда A_flag, B_flag подняты, поток A зациклен.  
B.2 вошёл в while.  
B.3 происходит, флаг опускается!  
A может спокойно выйти из цикла, но допустим этого пока не происходит.  
B.4 зацикливается и остаётся продолжать выполнение A.  
Но тогда всё продолжит работу, A_flag опустится и тд.  
Допустим B.2 зациклился перед A.2.  
Тогда A_flag, B_flag подняты, поток B вошёл в while.  
Если сейчас будет A.2 всё будет как в прошлом предположении.  
Если выполняется B.3, то флаг опускается и A пропустит while.  
То есть нужно снова поднять флаг в B.5, но while в B.4 мешает.  
Все варианты ведут к прошлому случаю.  

Допустим поток B зациклился в B.4.  
Для этого требуется A_flag == true.  
То есть чтобы A.5(единственное поднятие A_flag в коде) не выполнялось.  
Для этого нужно чтобы A зациклился в A.2.  
То есть B_flag == true.  
Перед A.2 следует A.1 - поднятие A_flag по concurrent execution trace.  
Перед A.2 следует B.1 - поднятие B_flag для зацикливания.  
Перед B.4 следует A.1 - поднятие A_flag для зацикливания.  
Перед B.4 следует B.3 - опускание B_flag по concurrent execution trace.  
Допустим A.2 было перед B.4.  
Тогда A_flag, B_flag подняты, поток A зациклен.  
B.4 может войти в цикл!  
Но перед ним необходимо поднять B_flag.  
A зациклен, иного выхода нет(  
Что приведёт нас к ситуации аналогичной рассматриваемой в самом начале ответа.
